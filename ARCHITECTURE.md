# Architecture

The bot uses Twilio Programmable Voice to place an outbound call to the assessment number and returns TwiML with `<Connect><Stream>`, which opens a bidirectional WebSocket to the FastAPI server. Twilio sends the called party's 8 kHz mulaw audio to `/media`; the server forwards those base64 audio chunks to an OpenAI Realtime voice-agent session. OpenAI returns generated patient speech as audio deltas, and the bridge streams those deltas back to Twilio as media messages. When the target agent starts speaking, the bridge sends Twilio `clear` and OpenAI `response.cancel` events to reduce overlap and improve turn-taking.

The caller behavior is hybrid scripted plus adaptive. Each scenario defines a patient profile, goal, facts, stressors, and success criteria. Those scenario details become the Realtime system instructions, so the caller can adapt naturally while still steering toward the intended test case. Twilio recordings are treated as the authoritative audio artifact, while Realtime transcript events are used to build speaker-labeled transcripts. The analyzer can draft findings from transcripts, but the submitted bug report is curated against the MP3 recordings and selected transcripts.
